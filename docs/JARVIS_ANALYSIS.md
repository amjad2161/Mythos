# JARVIS-Style Assistant Architectures: Comparative Analysis for Mythos

**Status:** Research survey, July 2026. Compares Mythos (this repository) against Microsoft JARVIS/HuggingGPT, the owner's agency-agents JARVIS layer, Leon AI, and (briefly) other prominent open "JARVIS-likes". Sources: project READMEs, repository trees, and the HuggingGPT paper. Direct fetches of arxiv.org/ar5iv and docs.getleon.ai returned HTTP 403; the HuggingGPT mechanisms cited here were cross-verified via the microsoft/JARVIS README, the NeurIPS 2023 proceedings listing, and secondary analyses, and Leon's transport was verified from its `package.json` and `server/src/core` tree rather than its docs site.

## 1. Architecture comparison

| Dimension | **Mythos** | **Microsoft JARVIS / HuggingGPT** | **agency-agents JARVIS layer** | **Leon AI** |
|---|---|---|---|---|
| **Planning / decomposition** | Orchestrator ("Agent Boss") decomposes goal into a `Workflow` of role-assigned steps; Phase B `--dynamic` uses a cheap routing LLM emitting strict JSON (`SchemaError` on violation, one verbatim re-prompt, deterministic fallback) | Stage 1 "task planning": LLM parses request into a JSON task list (`task`, `id`, `dep`, `args`) via prompt-engineered spec + demonstrations; no strict schema rejection, malformed plans degrade silently | `jarvis-core.md` meta-orchestrator persona: analyze → decompose → delegate → synthesize; runtime `planner.py` routes a message to one skill via LLM; delegation rules are prose, not schema | No goal decomposition; utterance → intent/skill match; newer versions add "agent mode" step-by-step planning per skill |
| **Routing / selection** | Role-based: deterministic keyword pre-filter narrows candidate roles, then LLM assigns steps to roles (each role = persona + tool subset); `backend_dev` always a candidate | Model-based: Stage 2 filters HF Hub models by task type, ranks by downloads, top-K into context, LLM picks per task — selects *models*, not agents | Skill-based: `SkillRegistry` loads 340+ markdown personas; planner picks a skill (`POST /api/plan` returns skill + rationale + candidates); "delegation hubs" for orchestrator personas | Intent/skill match with three modes: smart (auto), controlled (deterministic native skills), agent (planned workflows) |
| **Execution model** | Distributed swarm: `TaskPayload` JSON over RabbitMQ (`q.tasks.<role>`) to `WorkerAgent`s wrapping the single-agent core; concurrent dispatch of ready DAG branches; offline in-memory drivers | Central controller invokes selected models (HF inference endpoints / local / hybrid); independent tasks run in parallel; Stage 4 LLM synthesizes all outputs into one response | Single-process Claude tool-use loop (`executor.py`): file IO, shell allowlist, web fetch, skill delegation; multi-agent pipelines via CLI | Node server executes skill actions through bridges (Node.js/Python); layered Skills → Actions → Tools → Functions |
| **Verification / quality gate** | **Structural critic gate**: every worker result intercepted by `CriticAgent` (read/execute-only tools); mechanical `validation_command` first, LLM verdict fallback; fail-safe on no verdict; autonomous retry with verbatim `error_log`, max 3 attempts | None. Response generation trusts model outputs; no critic, retry-on-failure, or validation stage | None structurally; "never ship broken code" is persona prose; trust modes (`off`/`on-my-machine`/`yolo`) gate *permissions*, not output quality | None; skill output goes straight to the user |
| **Memory** | Data Matrix: Qdrant single collection, vectors + payload-embedded knowledge graph; verbatim content, **trust scores**, `navigate` = KNN + 1-hop traversal + trust-ordered fusion; durable per-goal task ledger | Ephemeral resource tracking within a request (outputs kept for dependent tasks); dialogue context in prompt; no persistent store | `MemoryStore` persists session transcripts to `~/.agency/sessions/`; profile + cross-session "lessons journal" | `memory-manager` + SQLite: layered memory (durable preferences, day-to-day context, recent discussion) |
| **Cost control** | Real token accounting (`LLMResponse.usage`) → cumulative budget enforced mid-run; prompt caching; `CostGovernor` sliding-hourly + per-run circuit breakers → structured FAILURE when tripped | Deployment-level only: `local`/`huggingface`/`hybrid` inference modes and minimal/full model sets; no token budgets | None visible; model choice via env config | Implicit: local models (`node-llama-cpp`) as a cost/privacy option; no budgets |
| **Comms (client ↔ system)** | HTTP control panel (`--serve`, 127.0.0.1:8642), goal submission + **polling** of live Task Ledger; internal comms are AMQP messages | Request/response HTTP POST (`/hugginggpt`, `/tasks`, `/results`); Gradio/web demo; no streaming | **SSE** streaming (`POST /api/run/stream`: `plan`, `text_delta`, `tool_use`, `tool_result`, `stop`, `error`, `done`) + **WebSocket** (`/ws/spatial`) for a bidirectional gesture-driven HUD | **socket.io WebSocket** (bidirectional, event-based) + Fastify HTTP; the reference JARVIS-like client experience |
| **Extensibility** | Roles (`roles.py` tool filtering), Markdown personas w/ strict frontmatter (`MYTHOS_PERSONA_DIR` overlay), `Tool` registration API, provider abstraction | New models appear automatically via HF Hub descriptions; task types are the fixed extension axis | Drop a markdown persona file → auto-discovered skill; MCP server config | Skills with structured manifests; toolkits, bridges (Node/Python), binaries |
| **Voice I/O** | **Output only**: `voice` role, `speak` tool → OpenAI-compatible TTS sidecar; no input | None (multimodal *content* tasks, not a voice interface) | None (text chat + webcam gestures) | **Full duplex**: dedicated `asr` and `tts` core modules, hotword detection, voice-first design |

**Briefly, other JARVIS-likes.** *Dipeshpal/Jarvis_AI* is a Python hobbyist assistant library: thin client (voice/text I/O via Google Speech or Whisper ASR, pyttsx3/gTTS output) calling a hosted server that does intent detection over ~22 command categories, with a simple callback mechanism for custom functions. Architecturally it is an intent-matcher, not an orchestrator; its only lesson for Mythos is the low-friction ASR-provider abstraction. *Open Interpreter* is a local code-execution agent (now a Rust-native coding-agent fork): an interactive loop that writes and runs code in a sandbox with human approval gates, extensible via MCP, skills, and hooks, exposing an Agent Client Protocol for editor embedding rather than an HTTP server. Its approval-gate UX and ACP editor integration are worth noting as alternative client surfaces, but it is a single-agent tool, not a multi-agent system.

## 2. What Mythos already does better

Verified against the compared systems' documentation and code trees — these capabilities are genuinely absent there:

1. **Structural critic quality gate.** In Mythos, workers *cannot* report to the orchestrator; `q.critic.review` is topologically unavoidable, the critic's tools are read/execute-only ("verifies, never fixes"), validation is mechanical-first, and unverifiable results fail safe. HuggingGPT has no verification stage at all — Stage 4 summarizes whatever the models returned. The agency-agents runtime has review *personas* but no interception point; Leon has none. Autonomous retry with the **verbatim** failure output in `error_log` (no human, no orchestrator in the loop) is unique among the four.
2. **Strict M2M schemas.** HuggingGPT's task JSON is prompt-format hopeful parsing; agency-agents and Leon exchange loosely-typed events. Mythos's `TaskPayload`/`StateUpdate` envelopes are strictly deserialized — unknown verbs/statuses or missing fields raise `SchemaError`, so a malformed message can never propagate silently. Combined with durable AMQP queues and manual acks, this is real machine-to-machine protocol discipline, not convention.
3. **Cost governance.** No compared system meters actual token spend. Mythos normalizes `usage` across providers, enforces `max_compute_tokens` as a *cumulative mid-run* budget, and trips hourly/per-run circuit breakers that produce structured refusals. HuggingGPT's "hybrid inference mode" and Leon's "run it locally" are deployment postures, not governance.
4. **Durable, externalized task ledger.** One stable matrix node per goal recording each step's role, objective, status, attempts, and summary — progress that survives the process. agency-agents persists chat transcripts and Leon persists conversational memory, but neither maintains machine-readable per-step execution state.
5. **Trust-scored, graph-linked, verbatim memory.** The Data Matrix's `trust_score` ordering (system instructions at 1.0 always outrank scraped content), `verbatim_required` reproduction, and KNN + edge-traversal `navigate` have no analogue in any compared system; the closest (Leon's layered memory) has no provenance or trust model.

Honest caveats: HuggingGPT's *model* selection over thousands of HF models is a broader routing space than Mythos's five roles; Leon's voice pipeline and the agency layer's streaming UX are ahead of Mythos's polling panel; agency-agents' 340-skill registry dwarfs Mythos's persona count.

## 3. What to adopt, ranked

### 3.1 Explicit resource dependencies between tasks (HuggingGPT) — **value: high, effort: low, fit: excellent**
Mythos already has `WorkflowStep.depends_on` and concurrent DAG dispatch, but a dependent task's *inputs* arrive only implicitly — the worker must re-`navigate` the matrix and hope semantic search surfaces its predecessor's artifact. HuggingGPT's `<resource>-task_id` placeholder makes the data flow explicit. Sketch:
- At dispatch time, the orchestrator resolves each completed dependency's `StateUpdate.result_pointers` and prepends them to the dependent `TaskPayload.task_parameters.context_pointers`. This is ~20 lines in `orchestrator.py`'s ready-task dispatch path and needs **no schema change** — `context_pointers` already exists and `navigate(seed_ids=...)` already consumes seeds.
- Let the decomposer emit `{{output:step_n}}` placeholders inside objectives; the orchestrator substitutes the artifact node UUID (never the content — pointers stay the currency) before dispatch, honoring the `literal=True` discipline.
- Result: dependency edges carry *data*, not just ordering, while everything still flows through the trust-scored matrix. This closes the one respect in which HuggingGPT's pipeline is genuinely tighter than Mythos's.

### 3.2 Real-time event streaming to clients — SSE first, WebSocket later (agency-agents, Leon) — **value: high, effort: moderate, fit: good**
The control panel polls; the agency server's SSE event vocabulary (`plan`, `tool_use`, `tool_result`, `error`, `done`) is the proven upgrade and maps almost 1:1 onto events Mythos already produces. Sketch:
- Add a `RunEventSink` the `SwarmRuntime` publishes to at existing choke points: ledger transitions (pending → dispatched → validated/failed), critic verdicts, retry loops, governor trips. These are the events users actually wait on.
- `orchestration/server.py` grows one endpoint, `GET /api/runs/{id}/events`, serving `text/event-stream` from a per-run in-memory queue. SSE keeps the panel dependency-free (stdlib `http.server` can hold a streaming response; the browser side is a ~10-line `EventSource` swap for the poll loop). Event shape: reuse the ledger step dict verbatim — no new schema.
- Defer WebSocket until there is client → run *interruption* (cancel, mid-run approval), which is the only thing SSE cannot do. Leon's socket.io and the `/ws/spatial` HUD show where that leads, but Mythos has no interactive-session semantics yet to justify it.

### 3.3 Skill/intent registry manifests (Leon, agency-agents) — **value: medium, effort: low-moderate, fit: good**
Routing today is a hardcoded keyword pre-filter (`prefilter_roles`) plus role/tool lists in the router prompt. Both compared systems make capabilities *declarative*. Sketch:
- Extend persona frontmatter (already strict) with `triggers:` (keywords), `examples:` (2–3 sample objectives), and `capabilities:` (one-line claims). `prefilter_roles` reads triggers instead of hardcoded lists; the decomposer prompt includes examples per candidate role — few-shot routing for free.
- Embed each role's `capabilities` into the matrix at startup; pre-filter by cosine similarity of goal → role manifests. This is the roadmap's "matrix-similarity pre-filter" and reuses `DataMatrix` wholesale.
- Do **not** adopt the 340-persona sprawl: agency-agents shows discovery scales, but Mythos's value is that every role carries an enforced tool subset and constraint profile; add roles only with their Tools API.

### 3.4 Voice input pipeline (ASR) (Leon, Dipeshpal) — **value: medium, effort: moderate, fit: good**
Mythos speaks but cannot listen; a JARVIS-style surface needs the other half. Sketch:
- Mirror the TTS pattern exactly: `MYTHOS_ASR_URL` pointing at any OpenAI-compatible `/v1/audio/transcriptions` sidecar (e.g. `faster-whisper-server`; add `--profile voice-in` to compose). One new tool module (`tools_asr.py`), symmetric with `tools_tts.py`.
- Control panel: `MediaRecorder` capture → POST audio to a panel endpoint → sidecar transcription → prefill the goal box for **human confirmation** before submission. Voice becomes an input method at the human boundary, not a bypass of it — consistent with "natural language exists only at the human boundary."
- Skip hotword/always-listening (Leon's hardest, most platform-specific machinery) until Phase C always-on autonomy gives it a purpose.

### 3.5 Streaming partial LLM output — **value: low for Mythos, effort: moderate-high, fit: weak — defer**
Honest assessment: token streaming (agency's `text_delta`) shines when a human reads one LLM's prose live. Mythos's LLM output is mostly machine-facing — strict JSON plans, worker tool-use loops, critic verdicts — where partial tokens are unparseable and worthless mid-flight. It would touch every provider backend and the `RetryingLLM`/usage-accounting wrappers for marginal UX on only the final conclusion. Adopt 3.2's *event-level* streaming first; revisit token streaming only if Mythos grows a direct human-chat surface.

## 4. What Mythos can offer back to a JARVIS federation

Mythos is a strong *executor tier* for any JARVIS-style front-end, which is precisely what the compared systems lack — they have interfaces and routing but weak execution guarantees.

- **HTTP goal API.** The control panel server already accepts goal submissions over HTTP on a serial run queue; hardening it into a small documented API (`POST /api/goals` → run id, `GET /api/runs/{id}` → ledger, plus §3.2's SSE stream) makes Mythos a drop-in backend. The agency-agents JARVIS could route "build/verify/execute" intents to Mythos exactly as it routes to a skill; Leon could wrap it as a skill whose action POSTs the goal and relays ledger events over socket.io. The roadmap's `callback_webhook` adapter completes the loop: the federation host registers a webhook and receives the terminal `StateUpdate` instead of polling.
- **Bus queues as an integration surface.** Because every boundary is a real AMQP message with a strict schema, an external orchestrator that can produce a valid `TaskPayload` can publish directly to `q.tasks.<role>` and receive `StateUpdate`s — inheriting the critic gate, retries, and cost governor without touching Mythos code. Strict deserialization means a foreign system cannot inject malformed work silently; `SchemaError` is a feature at a federation boundary. A HuggingGPT-style planner could target Mythos roles as if they were expert models.
- **The Data Matrix as shared federation memory.** Trust scores map naturally onto federation provenance: external systems write at low trust, Mythos-validated artifacts at higher trust, operator instructions at 1.0 — so shared memory degrades safely rather than poisoning context. `navigate`'s pointer-based fusion gives partners retrieval without bulk export, and verbatim nodes give them tamper-evident ground truth.
- **The bidirectional story.** Inbound: voice/chat front-ends (Leon, the agency HUD) capture intent and stream progress; Mythos supplies verified execution, budget enforcement, and a durable ledger. Outbound: Mythos's `researcher`/`navigator`/`voice` roles are callable services, and its `StateUpdate`s are structured enough for a front-end to narrate ("step 3 failed validation, retrying — attempt 2 of 3") — a far richer live experience than any compared system can offer *about its own* execution. The federation gets a JARVIS that not only answers, but proves its work.
